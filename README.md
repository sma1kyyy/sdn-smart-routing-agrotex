## Архитектура решения

### 1) Логическая модель сети

- **Рабочая сеть (Work):** `h1`, `h2` в подсети `10.0.1.0/24`.
- **Гостевая сеть (Guest):** `h3`, `h4` в подсети `10.0.2.0/24`.
- **Условный шлюз в Интернет:** `hgw` (`10.0.2.254/24`).
- **Коммутация:** три OVS-коммутатора (`s1`, `s2`, `s3`) с OpenFlow 1.3.
- **Контрольная плоскость:** ONOS как внешний SDN-контроллер.

### 2) Политика безопасности

Политика формализуется как deny-правило:

- **match:** IPv4, `src = 10.0.2.0/24`, `dst = 10.0.1.0/24`
- **action:** `DROP` (пустой treatment)
- **priority:** `50000` (выше обычной реактивной коммутации)
- **scope:** применяется ко всем активным устройствам ONOS

### 3) Почему это корректно

- Запрещается горизонтальное проникновение из гостевого сегмента в рабочий.
- Трафик гостя к шлюзу `10.0.2.254` не подпадает под deny-маску и остается разрешенным.
- Решение остаётся управляемым через Northbound REST API без установки Java-приложения в ONOS.

## Быстрый старт (для проверки пользователем)

> Ниже - минимальный маршрут запуска при уже установленном ONOS + Mininet.

### Шаг 1. Установить зависимости Python

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### Шаг 2. Запустить ONOS и убедиться, что REST доступен

```bash
curl -u onos:rocks [http://127.0.0.1:8181/onos/v1/devices](http://127.0.0.1:8181/onos/v1/devices)
```

### Шаг 3. Поднять Mininet-топологию

```bash
sudo python src/topology/agrotex_topology.py
```

### Шаг 4. Применить политику изоляции

```bash
python src/controller/guest_isolation.py apply
```

### Шаг 5. Проверка в Mininet CLI

```bash
mininet> h3 ping -c 3 h1
# ожидаемо: потери 100% (запрещено)

mininet> h3 ping -c 3 hgw
# ожидаемо: ответы есть (разрешено)

mininet> h1 ping -c 3 h2
# ожидаемо: ответы есть (рабочий сегмент функционирует)
```

### Шаг 6. Удалить политику (при необходимости)

```bash
python src/controller/guest_isolation.py remove
```

## Результаты демонстрации (что показывать преподавателю)

### Обязательный сценарий демонстрации

1. Запуск ONOS, вывод списка устройств (`/onos/v1/devices`).
2. Подъём топологии и проверка связности до применения политики.
3. Применение `guest_isolation.py apply`.
4. Демонстрация блокировки:
   - `h3 -> h1` и `h4 -> h2` не проходят.
5. Демонстрация разрешенного трафика:
   - `h3 -> hgw` и `h4 -> hgw` проходят.
6. Удаление политики (`remove`) и обратная проверка.

### Критерии успешности

- ONOS видит все коммутаторы (`available=true`).
- Правила изоляции появляются во flow-таблицах.
- Пинги из Guest в Work стабильно блокируются.
- Доступ к gateway host из Guest сохраняется.
