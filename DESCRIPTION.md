# __DESCRIPTION.md - Техническое описание проекта и выводы__

## 1) Краткое назначение проекта

Проект реализует учебный SDN-сценарий для компании AgroTex: **изоляцию гостевого сегмента от рабочей сети** в контролируемой среде Mininet под управлением ONOS.

Ключевая идея: политика безопасности задаётся программно через **Northbound REST API ONOS**, без ручной конфигурации каждого коммутатора.

---

## 2) Что реализовано

### 2.1 Топология сети

Реализована кастомная Mininet-топология с тремя коммутаторами и пятью хостами:

- `h1`, `h2` - рабочий сегмент (`10.0.1.0/24`);
- `h3`, `h4` - гостевой сегмент (`10.0.2.0/24`);
- `hgw` - условный шлюз/внешний ресурс (`10.0.2.254/24`);
- `s1`, `s2`, `s3` - OVS-коммутаторы (OpenFlow 1.3).

Такое разделение максимально прозрачно демонстрирует политику «запретить lateral movement из guest в work». 

### 2.2 Контроллерная логика

Скрипт `guest_isolation.py` реализует:

- обнаружение активных устройств через `/onos/v1/devices`;
- установку drop-flow правил через `/onos/v1/flows/{deviceId}`;
- удаление ранее установленных правил по match-критериям;
- параметризацию подсетей и приоритета через CLI-аргументы.

### 2.3 Проверки качества

Добавлен скрипт `scripts/run_policy_checks.sh`, который выполняет быстрые проверки:

- компиляция Python-скриптов (`py_compile`);
- синтаксис shell-скрипта;
- контроль обязательных файлов и структуры.

---

## 3) Почему решение соответствует учебной задаче

Требование задания для Команды 1: 

- запретить Guest → Work;
- сохранить Guest → Internet(условно через gateway).

Реализованная логика выполняет это буквально:

- deny-rule фильтрует только `10.0.2.0/24 -> 10.0.1.0/24`;
- трафик на `10.0.2.254` не блокируется;
- демонстрация проверяется командами `ping/iperf`.

Таким образом, решение функционально и методически закрывает поставленный критерий.

---

## 4) Основные решения

1. **REST API вместо Java-приложения ONOS**  
   Это снижает порог входа, ускоряет разработку и удобно для учебной демонстрации.

2. **Высокий приоритет deny-правил (`50000`)**  
   Обеспечивает преобладание политики безопасности над обычными реактивными маршрутами.

3. **Параметризуемые подсети**  
   Позволяет быстро адаптировать проект под другое адресное пространство без изменения исходников.

4. **Чёткое разделение ответственности файлов**  
   Топология, контроллерная логика, проверки и документация разнесены по структуре.

## 5) С какими проблемами можно столкнуться

### Проблема A: несогласованность версий ONOS / OpenFlow
**Симптом:** устройства видны нестабильно или не появляются правила.

**Решение:**
- принудительно использовать OpenFlow 1.3 на OVS;
- проверять подключение каждого коммутатора в ONOS (`available=true`).

### Проблема B: stale-состояние после нескольких прогонов
**Симптом:** «странные» потоки или маршруты после перезапусков.

**Решение:**
- `sudo mn -c` перед каждым чистым запуском;
- при необходимости restart `openvswitch`;
- повторно применить/снять политику.

### Проблема C: ошибки авторизации REST
**Симптом:** HTTP 401.

**Решение:**
- проверить креды `onos:rocks`;
- проверить, что REST-порт 8181 доступен локально.

### Проблема D: политика не блокирует трафик
**Симптом:** `h3 -> h1` продолжает пинговаться.

**Решение:**
- проверить фактические IP и маски хостов;
- проверить, что flow действительно добавился на нужные устройства;
- повысить приоритет deny-rule, если в среде есть конкурирующие правила.
